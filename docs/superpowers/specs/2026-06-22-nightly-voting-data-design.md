# 次日投票数据与智能表格功能设计

## 1. 文档状态

- 日期：2026-06-22
- 状态：已确认，待实施计划
- 产品：`boatraceCal`
- 视觉方向：方案 3“智能表格”
- 适用范围：独立采集、数据治理、两阶段概率预测、价值策略、纸面推荐、赛后结算及操作界面

## 2. 背景与目标

`boatraceCal` 每天晚上生成第二天的纸面投票预案，并在比赛当天按赛前数据和赔率更新为正式决策。系统必须独立开发、测试和运行，不依赖旧程序、旧项目目录或旧 Excel 的内部结构。

旧 Excel 只用于理解长清单的信息密度。旧系统的结果匹配规则作为迁移参考，用于保留号码归一化、五种票型结算和聚合报表语义，不成为新系统的运行依赖。

系统不承诺盈利、不自动下注、不管理真实资金。所有输出必须区分预案与最终决策，并可追溯到数据、特征、模型和策略版本。

## 3. 范围

### 3.1 首个可运行研究 MVP

- 仅使用日本 BOAT RACE 官方公开来源。
- 项目自身负责历史数据回补、赛前页面抓取、赔率定时采集、失败重试和采集审计。
- 首轮试点一个可配置场地，回补最近五个完整赛季，再扩展到全部场地。
- 保存开赛前 30、15、10、5 分钟的赔率快照。
- 建立夜间预案模型和比赛日最终模型。
- 首版模型只生成三连单 120 种有序组合的概率。
- 建立独立价值策略、纸面模拟、结果匹配与结算。
- 提供 CLI、静态 HTML 数据质量报告、FastAPI/OpenAPI 和 React 智能表格。
- 支持 `.xlsx` 导出，不覆盖原文件。

### 3.2 非目标

- 账号登录、自动下注、资金托管和真实投注执行。
- 马丁格尔、亏损追投或稳定盈利承诺。
- 第一版自动推荐三连复、二连单、二连复或扩连复。
- 兼容旧 `.xls`、COM 或 ACE OLEDB 环境。
- Airflow、Celery、微服务或外部分布式队列。

## 4. 核心业务流程

### 4.1 清单状态

```text
PREPARING
→ PREPLAN_READY
→ WAITING_PRERACE_DATA
→ ODDS_REFRESHING
→ FINAL_READY
→ CONFIRMED / EXPIRED
```

状态必须显示中文名称，并保留稳定英文代码供 API、测试和日志使用。

### 4.2 夜间预案

系统在每天晚上为下一业务日期执行：

1. 获取次日赛程、选手、艇位和已公开机艇数据。
2. 执行数据完整性与时间安全检查。
3. 使用夜间模型生成三连单 120 种组合的基础概率。
4. 输出候选组合、风险占位和模拟预算预案。
5. 将清单标记为 `PREPLAN_READY`，不得展示虚构赔率或 EV。

夜间预案中尚不可获得的赔率、展示数据和 EV 必须显示“等待赛前数据”，不能以零、最终赔付或推算值代替。

### 4.3 比赛日更新

系统在 T-30、T-15、T-10、T-5 采集合法赔率和可用赛前数据：

1. 保存带 `observed_at` 和 `available_at` 的快照。
2. 重新生成时间安全特征。
3. 使用比赛日模型输出校准概率。
4. 计算市场隐含概率、EV、保守 EV 和置信度。
5. 在 T-10 冻结正式 `SELECT/PASS` 决策。
6. T-5 只生成变化警报，不静默覆盖已经确认的版本。

数据缺失、赔率陈旧、模型不确定或风险超限时必须输出 `PASS`。

### 4.4 用户审核

用户可以：

- 查看次日全部场地、场次和候选组合。
- 按场地、清单状态、EV、置信度和数据质量筛选。
- 查看单个组合的概率、赔率、EV、数据新鲜度和推荐/拒绝原因。
- 调整纸面模拟单位、改为 `PASS` 或添加备注。
- 确认一个不可变清单版本并导出 Excel。

“确认明日清单”仅冻结纸面模拟版本，不产生任何外部投注行为。

## 5. 系统架构

```text
官方公开页面
    ↓
任务规划与账本
    ↓
限速抓取 → 版本化解析 → 契约校验
    ↓
标准化数据仓
    ├─→ 时间安全特征 → 夜间模型 → 次日预案
    ├─→ 赛前快照     → 最终模型 → 校准概率
    └─→ 赛后结果     → 结算引擎 → 日报/回测
                              ↓
                       FastAPI / OpenAPI
                              ↓
                         React 智能表格
```

采用 Python 模块化单体。模块职责如下：

| 模块 | 职责 |
|---|---|
| `ingestion` | 官方页面抓取、限速、重试和解析器版本管理 |
| `jobs` | 历史回补、实时快照、检查点和任务账本 |
| `validation` | 主键、范围、完整性、时效与时间穿越检查 |
| `storage` | DuckDB 目录数据与 Parquet 数据快照 |
| `features` | 分别定义夜间特征和赛前特征 |
| `models` | 基线模型、候选模型、训练、校准和推理 |
| `strategies` | 隐含概率、EV、置信度和风险门禁 |
| `recommendations` | 预案、最终清单、用户审核和不可变版本 |
| `settlement` | 五种票型的标准化匹配与纸面结算 |
| `reporting` | HTML 质量报告、Excel 清单和日报 |
| `api` | 稳定的 OpenAPI 契约 |
| `web` | React 智能表格，不直接访问数据库 |

边界约束：

- 抓取器不得直接调用模型。
- 模型只估计概率，不决定投注单位。
- 策略不得修改模型输出，只生成独立决策记录。
- 界面不得覆盖原始预测，只能新增审核、备注和模拟单位调整。
- 结算数据不得进入同一比赛的赛前特征。
- 外部网站、旧程序和旧 Excel 均不得成为运行依赖。

## 6. 采集任务设计

### 6.1 任务幂等键

```text
source + venue + race_date + race_no + data_type + snapshot_target
```

`snapshot_target` 对历史静态数据可以为空，对实时赔率使用 `T30`、`T15`、`T10`、`T05`。

### 6.2 任务状态

```text
PENDING → RUNNING → SUCCEEDED
                  ↘ RETRY_WAIT → RUNNING
                  ↘ FAILED
                  ↘ SKIPPED
```

每个任务至少记录尝试次数、最后错误代码、下一重试时间、检查点、解析器版本和数据产物标识。

### 6.3 历史回补

- 按场地、日期和数据类型分页执行。
- 每个成功日期写入检查点，支持暂停、恢复和幂等重跑。
- 单日失败不阻塞整个日期范围。
- 官方不提供的数据标记 `SOURCE_UNAVAILABLE`，不无限重试。
- 不使用最终赔付近似历史赔率。

### 6.4 实时采集

- Windows 任务计划程序只负责周期性调用 CLI。
- CLI 根据当日赛程生成到期快照任务。
- 超过有效时间窗口后不补造赔率快照，记录 `MISSED_WINDOW`。
- 同一快照重跑不得产生重复业务记录。

### 6.5 重试与限速

- 网络超时和临时 5xx 使用指数退避与随机抖动。
- 页面尚未发布时只在允许窗口内延迟重试。
- 解析失败或页面结构变化时立即隔离，停止高频请求。
- 达到最大次数后进入人工检查队列，并在质量报告置顶。
- 正式抓取前必须现场确认官方条款、robots 规则、访问入口和可接受频率；未完成确认不得启用持续采集。

## 7. 数据留存与时间安全

### 7.1 原始响应

- 成功响应不长期保存正文。
- 保存 URL、抓取时间、HTTP 状态码、内容哈希和解析器版本。
- 失败或解析异常的原始响应隔离保存 30 天。
- 清理任务记录删除文件数、大小、执行时间和失败原因。

### 7.2 标准化数据

所有赛前记录必须包含：

- `observed_at`
- `available_at`
- `source`
- `source_hash`
- `parser_version`

任何预测只能读取 `available_at <= prediction_as_of` 的记录。后补数据不得伪装成预测时点已经可用的数据。

## 8. 模型与策略设计

### 8.1 夜间模型

只使用前一晚已公开的数据，例如：

- 场地和艇位历史表现。
- 选手全国/当地成绩及平均 ST。
- 机艇信息与预测时点之前的历史表现。
- 比赛级别、场次和季节特征。

夜间模型输出 120 种三连单组合概率与初步候选，不输出依赖实时赔率的最终 EV。

### 8.2 比赛日模型

在夜间特征基础上增加：

- 展示航行、展示 ST 和进场队形。
- 天气、水温、风浪。
- 合法赔率快照及变化。
- 数据时效与完整性指标。

基线使用 scikit-learn Logistic Regression，候选模型使用 HistGradientBoosting。模型在独立时间验证窗口进行概率校准。夜间模型与比赛日模型分别训练和评价，不以缺失值强行共用同一个模型。

### 8.3 策略层

```text
校准概率 + 当时赔率
→ 市场隐含概率
→ EV 与保守 EV
→ 数据质量门禁
→ 风险限制
→ SELECT / PASS
```

首版使用固定模拟单位。夜间预案可以提供建议区间，最终单位只在 T-10 正式决策生成后确定。

## 9. 结果匹配与旧格式兼容

### 9.1 核心匹配键

```text
race_id + bet_type + normalized_combination
```

- 三连单、二连单保留号码顺序。
- 三连复、二连复、扩连复按号码排序后构造匹配键。
- 推荐必须绑定具体场次。
- 赔付与结算记录只追加，不覆盖推荐。

### 9.2 票种边界

- 模型和自动推荐首版只处理三连单。
- 结算数据结构兼容三连单、三连复、二连单、二连复和扩连复。
- 其他票种可以导入、匹配和展示，但不自动生成推荐。

### 9.3 旧 Excel 适配器

旧字段映射如下：

```text
競賽場   → 场地、场次或聚合范围
競賽類型 → 标准 bet_type
警報     → 标准号码组合
```

- 没有场次后缀的记录标记为场地日聚合，展开到具体场次做兼容匹配。
- 旧代码 25–28 的固定场地组合、29 的半天组合和 30 的全天组合改为保存的报表视图，不作为真实场地代码。
- 旧格式导入只用于历史核验和兼容报表，不进入模型训练。

### 9.4 Excel 输出

默认生成新的 `.xlsx` 文件，包含：

- `次日预案`
- `最终清单`
- `结算明细`
- `payouts`
- `日报汇总`

第一版不支持 `.xls` 和 COM/ACE OLEDB。

## 10. 智能表格交互设计

### 10.1 视觉方向

采用方案 3“智能表格”：以宽表格为主要工作面，保留旧 Excel 的快速扫描能力，同时增强分组、状态、风险和解释层级。

### 10.2 顶部状态栏

展示业务日期、生成状态、数据新鲜度、场地/场次/候选数量、模拟预算、风险警告、Excel 导出和“确认明日清单”。

### 10.3 主表格

按场地、场次和候选组合分组，冻结表头与首列。核心列：

```text
场地 / 场次 / 开赛时间 / 组合
模型概率 / 市场赔率 / 隐含概率
EV / 保守EV / 置信度
建议单位 / 决策 / 审核状态 / 备注
```

夜间不可获得的数据必须显示“等待赛前数据”。

### 10.4 行级详情

选择一行后，在底部展开分析面板，展示：

- 六艇概率构成。
- 模型概率与市场概率比较。
- 主要支持因素。
- 数据新鲜度。
- 备选组合及拒绝原因。
- 数据、模型和策略版本。
- 调整模拟单位、改为 `PASS` 和备注操作。

### 10.5 更新与批量操作

- 新快照到达后高亮变化字段，不打乱当前排序。
- 显示概率、赔率和决策的前后差异。
- 已确认版本不被静默覆盖。
- 批量操作仅允许“设为 PASS”和“确认审核”，不允许批量强制 `SELECT`。

## 11. API 边界

FastAPI 通过 OpenAPI 暴露稳定契约，至少覆盖：

- 查询业务日期和清单状态。
- 查询场地、场次和候选组合。
- 查询单个组合解释和版本信息。
- 新增审核记录、备注和模拟单位调整。
- 确认不可变清单版本。
- 触发 Excel 导出并查询产物状态。

API 不暴露直接改写预测、赔率、赔付或原始标准化数据的接口。

## 12. 错误处理

错误必须使用稳定代码，并按数据、采集、解析、模型、策略和操作分层。示例：

```text
SOURCE_UNAVAILABLE
MISSED_WINDOW
FETCH_TIMEOUT
RATE_LIMITED
PARSE_SCHEMA_CHANGED
DQ_MISSING_ENTRY
DQ_STALE_ODDS
DQ_INCOMPLETE_ODDS
DQ_TIME_LEAK_RISK
MODEL_NOT_READY
CALIBRATION_INVALID
STRATEGY_RISK_LIMIT
VERSION_CONFLICT
```

用户界面显示可理解的中文说明，同时保留错误代码供审计。任何失败不得静默转换为默认推荐。

## 13. 测试设计

### 13.1 单元测试

- 场地、场次、票种和号码标准化。
- 顺序型与组合型匹配。
- 五种票型结算。
- 任务幂等键、状态转换与重试上限。
- 时间安全过滤。
- EV、保守 EV 和 `SELECT/PASS`。
- 预案与最终清单版本冻结。

### 13.2 集成测试

使用项目内固定 HTML 夹具，不依赖官网实时状态：

- 页面解析到标准化数据。
- 页面结构变化触发解析异常。
- 历史回补中断恢复。
- 四个赔率截点调度。
- 推荐、结算、日报和 Excel 导出。
- FastAPI 响应符合 OpenAPI 契约。

### 13.3 模型测试

- 训练、验证和测试按时间隔离。
- 同一比赛不得跨数据集。
- 概率和、Log Loss、Brier 和 ECE 可复现。
- 与艇位频率和市场基线在相同窗口比较。
- 夜间模型与比赛日模型分别报告。
- 未优于基线时不得宣称模型有效。

### 13.4 界面端到端测试

覆盖日期切换、场地筛选、排序、详情展开、单位调整、设为 `PASS`、版本确认和 Excel 导出。

## 14. 试点验收标准

- 单场地最近五个完整赛季回补完成，并可断点续传。
- 连续 14 天实时采集稳定运行。
- T-30、T-15、T-10、T-5 赔率快照达成率不低于 95%。
- 核心主键重复数为 0。
- 所有记录具有来源、采集时间、可用时间和解析器版本。
- 时间穿越检查全部通过。
- 缺失、重试、异常与清理行为可从报告追踪。
- 相同数据、配置和随机种子可复现主要结果。
- 界面不把夜间预案展示为最终推荐。

## 15. 已知风险与实施门禁

| 风险 | 设计响应 |
|---|---|
| 官方页面结构和入口尚未现场确认 | 实施前完成只读页面盘点，解析器以夹具驱动开发 |
| 官方访问频率及条款尚未现场确认 | 未完成条款、robots 和频率核验前不得启用持续采集 |
| 历史赔率可能不可获得 | 明确登记缺失，不使用最终赔付或赛后数据近似 |
| 不长期保存成功原始响应降低重解析能力 | 保存内容哈希和解析器版本；异常响应保留 30 天 |
| 夜间信息不完整 | 使用独立夜间模型，预案与最终决策分离 |
| 高赔率小样本导致回测失真 | 使用校准、Bootstrap 区间、切片报告和保守 EV |
| 旧格式语义复杂 | 通过边界适配器兼容，不污染核心领域模型 |

## 16. 决策摘要

- 采用带任务账本的模块化采集流水线。
- 官方公开来源是唯一数据来源，旧项目和旧程序不作为依赖。
- 先单场地试点，再扩展全部场地。
- 夜间预案与 T-10 最终决策使用两套独立模型。
- 模型只生成概率，策略独立决定 `SELECT/PASS`。
- 模型首版只推荐三连单，结算结构兼容五种票型。
- 结构化数据长期保存；异常原始响应保留 30 天。
- UI 采用高密度智能表格，确认仅冻结纸面模拟版本。

## 17. 实施拆分

本规格描述完整研究 MVP，但实现必须按以下独立阶段交付，不能一次性铺开：

1. **工程与领域契约**：项目骨架、核心实体、配置、版本字段和测试基线。
2. **采集与数据治理**：任务账本、单场地历史回补、实时快照、解析器、质量报告和时间安全检查。
3. **概率与策略闭环**：夜间模型、比赛日模型、概率校准、价值策略、回测和不可变推荐版本。
4. **结算与兼容输出**：五票型匹配结构、旧 Excel 导入适配器、纸面结算和 `.xlsx` 报告。
5. **API 与智能表格**：FastAPI/OpenAPI、React 方案 3 界面和端到端交互测试。
6. **试点运行与扩展**：单场地五年回补、14 天实时验证、验收报告，再决定是否扩展全部场地。

每一阶段都必须在测试通过、产物可复现且已知错误为零后，才能进入下一阶段。首份实施计划只覆盖第 1 阶段，避免把尚未验证的官网结构和模型假设扩散到整个代码库。
