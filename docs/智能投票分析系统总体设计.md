# 智能投票分析系统总体设计

## 1. 文档目的

本文给出 `boatraceCal` 的可实施设计。系统基于 BOAT RACE 历史及赛前数据，估计比赛结果概率，并结合下注时赔率筛选可能具有价值的候选组合。

系统目标不是预测“必中号码”，而是回答四个问题：

1. 各比赛结果在当前信息下的合理概率是多少？
2. 市场赔率是否充分补偿了该结果的风险？
3. 在什么条件下应当明确放弃投注？
4. 策略在未见过的历史时期是否具有稳定、可复现的表现？

## 2. 范围与非目标

### 2.1 第一阶段范围

- 通过本项目自有、版本化的数据契约导入比赛结果和赔付数据，数据来源位置可配置且不依赖其他项目。
- 采集或导入赛前节目表、选手、艇位、机艇、展示、气象和赔率快照。
- 构建三连单 120 种有序组合的概率预测。
- 进行概率校准、价值计算、候选筛选和离线回测。
- 输出单场分析、候选组合、弃权原因和策略日报。
- 支持纸面模拟，不涉及真实资金。

### 2.2 非目标

- 不承诺提高盈利或实现稳定盈利。
- 不使用最终赔付推导下注时可用赔率。
- 不通过倍投掩盖负期望策略。
- 第一阶段不实现账号登录、自动下注和资金托管。
- 不在数据量与基线不足时引入复杂深度学习架构。

## 3. 成功标准

系统成功不以单一命中率判断。离线验收至少满足：

| 维度 | 验收口径 |
|---|---|
| 数据完整性 | 核心比赛主键重复率为 0；关键赛前字段缺失率有报表 |
| 因果正确性 | 特征 `available_at` 不晚于预测截点；泄漏检查全部通过 |
| 概率质量 | 在最终测试集上优于艇位频率基线的 Log Loss 与 Brier Score |
| 校准能力 | 预测概率分桶后给出 ECE 和可靠性曲线 |
| 策略表现 | 同时报告回收率、命中率、覆盖率、最大回撤和置信区间 |
| 稳定性 | 分月份、场地、赔率区间均展示结果，不隐藏亏损切片 |
| 可复现性 | 相同数据快照、配置和种子可复现主要指标 |

回收率达到 100% 以上只能视为研究信号，不能单凭一次回测宣称策略有效。是否进入小额实盘阶段，需要独立的纸面模拟验证。

## 4. 总体架构

```mermaid
flowchart LR
    A["外部文件或数据 API"] --> B["独立导入适配器"]
    B --> C["采集元数据与异常隔离区"]
    D["赛前数据与赔率快照"] --> E["清洗与数据契约校验"]
    C --> E
    E --> F["标准化比赛数据库"]
    F --> G["时间安全特征构建"]
    G --> H["概率模型训练与校准"]
    H --> I["模型注册与推理"]
    F --> J["回测结算引擎"]
    I --> K["价值评估与策略过滤"]
    K --> J
    K --> L["单场分析与纸面推荐"]
    J --> M["指标、报告与风险审计"]
```

### 4.1 分层职责

| 层 | 职责 | 关键产物 |
|---|---|---|
| 数据接入 | 导入、抓取、去重、记录来源 | 采集元数据、内容哈希、异常响应样本 |
| 数据标准化 | 统一比赛、选手、赔率和结果契约 | 规范化表、质量报告 |
| 特征 | 只生成预测时点可用的统计量 | 特征快照、特征清单 |
| 模型 | 概率预测、校准和版本管理 | 模型文件、训练报告 |
| 策略 | 比较概率与赔率，决定入选或弃权 | 候选组合、拒绝原因 |
| 回测 | 仿真下注、结算、资金曲线 | 回测明细、风险指标 |
| 应用 | 查询、可视化、日报和模拟 | API、CLI、GUI/报告 |

## 5. 核心业务流程

### 5.1 离线训练

1. 锁定原始数据快照与预测截点，例如开赛前 10 分钟。
2. 对每场比赛按 `race_id` 生成赛前特征。
3. 按时间划分训练、验证和最终测试窗口。
4. 训练基线模型和候选模型。
5. 在验证集调参并进行概率校准。
6. 冻结模型、策略阈值和测试规则。
7. 仅一次性运行最终测试并登记结果。

### 5.2 单场推理

1. 校验比赛状态及赛前数据完整性。
2. 按指定预测截点选择最后一份合法赔率快照。
3. 生成特征并输出 120 种组合概率。
4. 将赔率换算为市场隐含概率并考虑庄家/返还率影响。
5. 计算每个组合的期望值与不确定性。
6. 通过最低优势、赔率区间、数据质量和风险规则过滤。
7. 输出有限候选；没有合格候选则明确弃权。

### 5.3 赛后结算

1. 通过独立导入适配器读取符合版本化契约的官方结果与 `payouts`。
2. 以 `race_id + bet_type + combination` 唯一匹配。
3. 保存推荐时看到的赔率与最终结算赔付，二者不得覆盖。
4. 更新模拟资金曲线和分层指标。
5. 监控数据漂移、校准漂移和策略异常。

## 6. 概率与价值定义

设组合 `c` 的模型概率为 `p(c)`，下注时十进制赔率为 `o(c)`，每单位投注的理论期望收益为：

```text
EV(c) = p(c) × o(c) - 1
```

实际入选不能只依赖 `EV > 0`，还应满足：

- 概率模型已校准；
- 数据完整性达到最低门槛；
- 赔率快照足够接近预测截点；
- 保守概率下界计算出的 EV 仍高于策略阈值；
- 单场、单日和相关组合风险没有超限。

## 7. 推荐输出契约

每次分析输出以下字段：

| 字段 | 含义 |
|---|---|
| `analysis_id` | 本次分析唯一标识 |
| `race_id` | 比赛唯一标识 |
| `as_of` | 数据截点 |
| `model_version` | 模型版本 |
| `strategy_version` | 策略版本 |
| `combination` | 三连单组合 |
| `model_probability` | 校准后概率 |
| `market_odds` | 当时赔率 |
| `implied_probability` | 赔率隐含概率 |
| `expected_value` | 单位期望收益 |
| `confidence` | 数据与模型置信等级 |
| `stake_units` | 模拟投注单位，首版默认固定 0 或 1 |
| `decision` | `SELECT` 或 `PASS` |
| `reason_codes` | 入选或弃权原因代码 |

## 8. 技术选型原则

- 语言优先 Python，以统一数据、模型、回测和接口层的实现生态。
- 本地研究阶段使用 Parquet 数据快照与 DuckDB；达到经测量的规模瓶颈后再评估迁移。
- 表格建模优先 Logistic Regression 与梯度提升树，并使用独立校准器。
- 配置、数据版本、模型版本、策略版本分开管理。
- CLI 优先于 GUI，先保证批处理、复现和自动测试，再建设界面。

目标技术路线已经锁定，但依赖尚未安装；实施时仍须获得用户对新增依赖的明确同意，并按官方兼容矩阵固定版本。

## 9. 关键风险

| 风险 | 后果 | 缓解措施 |
|---|---|---|
| 数据泄漏 | 回测虚高，实盘失效 | `available_at`、时间切分、自动泄漏测试 |
| 赔率时点错误 | 价值判断不可执行 | 保存多时点快照，按预测截点回放 |
| 过拟合 | 局部历史优秀、未来失效 | 简单基线、滚动回测、最终测试封存 |
| 市场与规则变化 | 模型漂移 | 分期监控、重训门槛、版本化规则 |
| 小样本高赔率 | ROI 被少数偶然命中主导 | Bootstrap 区间、最低样本量、切片报告 |
| 追求命中率 | 高频低赔率但长期亏损 | 以回收率和风险调整指标为主 |
| 抓取不稳定 | 特征缺失或错位 | 采集元数据、异常响应隔离、幂等重试、数据质量门禁 |

## 10. 架构决策记录

### 决策

采用“概率模型 + 独立价值策略 + 时间滚动回测”的模块化单体架构，先完成离线研究与纸面模拟。

### 驱动因素

1. 防止概率预测、资金策略和结算逻辑互相污染。
2. 在有限团队和本地环境中保持可实施与可复现。
3. 给未来 API、GUI 或自动化预留清晰边界，但不提前建设分布式系统。

### 备选方案

- 规则打分系统：开发快、可解释，但难以校准概率，作为基线保留。
- 端到端深度学习：表达能力强，但数据量、解释性和泄漏治理成本过高，第一阶段不采用。
- 微服务架构：部署独立，但当前规模下运维成本大于收益，不采用。

### 后果

- 正面：模块边界清晰，易测试，模型可替换，策略可独立回测。
- 负面：需要严格维护数据时间语义和多版本产物，前期数据工程工作量较大。

### 后续决策

- 确认首个预测截点和允许的数据延迟。
- 确认首版是否仅覆盖三连单。
- 数据盘点后再确定存储引擎及具体机器学习依赖。
