# 数据与特征设计

## 1. 设计原则

- 成功抓取长期保存采集元数据、内容哈希和不可变的标准化记录，不长期保存响应正文。
- 抓取失败或解析异常的原始响应进入隔离区并保留 30 天，用于诊断页面结构变化。
- 所有记录包含来源时间 `observed_at` 和业务可用时间 `available_at`。
- 任何预测只能读取 `available_at <= prediction_as_of` 的记录。
- 比赛主键统一为 `race_id = YYYYMMDD-JCD-RNO`。
- 赛后字段仅可用于标签、结算和分析，不得进入赛前特征。
- 数据质量不足时应停止推荐，而不是用默认值制造确定性。

## 2. 核心实体

### 2.1 `race`

| 字段 | 示例 | 说明 |
|---|---|---|
| `race_id` | `20260510-05-01` | 比赛唯一标识 |
| `race_date` | `2026-05-10` | 比赛日期 |
| `jcd` | `05` | 场地代码 |
| `rno` | `1` | 场次 |
| `scheduled_at` | 时间戳 | 计划开赛时间 |
| `grade` | 字符串 | 比赛级别 |
| `status` | 枚举 | 计划、截止、完成、取消 |

### 2.2 `race_entry`

一场比赛通常六条参赛记录，建议主键为 `race_id + lane`。

主要字段：选手编号、艇位、等级、体重、全国/当地胜率、二连率、三连率、平均 ST、调整重量、机号、艇号及公布时间。

### 2.3 `exhibition_snapshot`

保存展示航行、展示 ST、进场队形等赛前信息。每次更新均追加快照，不覆盖旧值。

### 2.4 `weather_snapshot`

保存气温、水温、风向、风速、波高、天气和观测时间。

### 2.5 `odds_snapshot`

| 字段 | 说明 |
|---|---|
| `race_id` | 比赛标识 |
| `bet_type` | 票种 |
| `combination` | 规范化组合 |
| `odds` | 当时赔率 |
| `observed_at` | 抓取时间 |
| `available_at` | 系统可使用时间 |
| `source` | 数据来源 |

唯一性建议：`race_id + bet_type + combination + observed_at`。

### 2.6 `race_result` 与 `payout`

结果表保存最终名次、起步、违规和比赛状态；赔付表使用本项目拥有并版本化的标准字段，外部来源由适配器映射到该契约：

```text
jcd, jcd_name, hd, rno, rno_name,
bet_type, number_set, number_set_spaced,
payout_yen, popularity
```

这些字段必须标注为 `post_race_only`。

### 2.7 `prediction`、`recommendation` 与 `settlement`

- `prediction`：模型对所有候选组合的概率。
- `recommendation`：策略基于概率、赔率和风险规则做出的选择或弃权。
- `settlement`：使用官方结果计算模拟盈亏，保存结算规则版本。

三类记录均保留数据、特征、模型和策略版本。

## 3. 特征分组

### 3.1 艇位与场地

- 场地各艇位历史一着率、前三率。
- 场地 × 月份、风向、风速区间的艇位表现。
- 当前场次编号、比赛级别和昼夜类型。

### 3.2 选手能力

- 截点前公开的全国及当地统计。
- 过去 N 场的指数加权表现，只使用发生在当前比赛之前的记录。
- 不同艇位的胜率、平均 ST 和稳定性。
- 事故、犯规、迟发等状态，但必须依据当时已公开信息。

### 3.3 机艇表现

- 发动机及艇体的历史二连率、三连率。
- 当前赛事期间的滚动表现。
- 机艇指标相对本场六人的排名和差值。

### 3.4 展示与环境

- 展示航行时间及场内标准化排名。
- 展示 ST、进场队形变化。
- 风、浪、温度及其与艇位的交叉特征。

### 3.5 市场特征

- 预测截点最近赔率。
- 多个合法截点的赔率变化率。
- 市场隐含概率、热度排名和集中度。

市场特征用于价值判断时必须谨慎：可训练“含市场”和“不含市场”两个模型，比较增益并检查是否只是复制热门排序。

## 4. 标签设计

首版三连单可采用两条路线：

1. 直接多分类：每场对应 120 种排列中的一个类别。
2. 分阶段排序：分别估计名次或成对胜出概率，再生成联合概率。

建议先实现直接多分类基线，同时保留分阶段模型作为对照。标签只来源于官方最终结果，取消、无效或无法可靠结算的比赛应按明确规则排除并记录原因。

## 5. 时间安全规范

每个特征必须登记：

| 元数据 | 说明 |
|---|---|
| `feature_name` | 特征名 |
| `source_table` | 来源表 |
| `event_time` | 事件发生时间 |
| `available_at` | 可用于预测的时间 |
| `lookback` | 历史窗口 |
| `null_policy` | 缺失处理 |
| `owner` | 维护模块 |

自动检查至少覆盖：

- 特征来源是否包含比赛结果或赔付。
- 聚合窗口是否错误包含当前比赛。
- 赔率快照是否晚于预测截点。
- 同一比赛是否跨训练和测试集合。
- 回补数据是否伪装成当时已知信息。

## 6. 数据质量门禁

单场进入推理前执行：

- 六个艇位是否齐全且唯一。
- 选手编号与艇位是否可解析。
- 比赛时间、场地和场次是否一致。
- 赔率 120 个组合的覆盖率是否达标。
- 展示与天气快照是否在允许的时效范围内。
- 关键数值是否在合理区间。

门禁失败时输出标准原因，例如：

```text
DQ_MISSING_ENTRY
DQ_STALE_ODDS
DQ_INCOMPLETE_ODDS
DQ_TIME_LEAK_RISK
DQ_RESULT_CONFLICT
```

## 7. 数据集版本

每个数据集清单至少包含：

- 数据时间范围和比赛数量；
- 原始文件哈希或快照标识；
- 过滤规则和排除数量；
- 特征定义版本；
- 预测截点；
- 生成代码版本；
- 数据质量摘要。

禁止只保存最终训练矩阵而丢失其构建依据。
