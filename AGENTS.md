# AGENTS.md

本文件约束 Codex 及其他自动化开发代理在 `boatraceCal` 项目中的工作方式。

## 项目定位

`boatraceCal` 是日本 BOAT RACE 的智能分析与决策支持系统。系统消费历史及赛前数据，输出概率、价值评分、候选组合、风险提示和回测报告。

本项目不承诺盈利，不提供“必中”结论；默认不实现自动下单。任何推荐都必须可追溯到数据版本、模型版本和策略版本。

## 核心目标

- 使用下注时可获得的数据估计比赛结果概率。
- 比较模型概率与市场隐含概率，识别可能的正期望候选。
- 通过严格的时间序列回测评价命中率、回收率和风险。
- 提供可解释、可复现、可审计的分析结果。

## 语言与沟通

- 全程使用中文输出，代码标识符遵循项目既有英文命名。
- 修改前用三点说明：改什么、为什么、风险或副作用。
- 最终汇报包含：变更文件、关键修改、验证证据、剩余风险。
- 不把推测写成事实；无法验证的结论明确标注假设。

## 开发原则

1. 读先于改：先检查相关代码、配置、数据契约和测试。
2. 最小变更：不顺手重构无关模块，不新增未要求的功能。
3. 测试先行：缺陷修复先补回归测试；核心算法先定义可验证样例。
4. 时间因果：任何特征必须记录 `available_at`，禁止使用预测时点之后的数据。
5. 可复现：训练、回测和推荐必须固定数据集快照、随机种子及版本号。
6. 可解释：推荐必须同时输出概率、赔率、期望值、置信度和拒绝原因。
7. 默认弃权：没有满足阈值的候选时输出“不建议投注”，不得强制凑单。
8. 无新增依赖：除非用户明确同意，不引入新的第三方依赖。

## 技术路线约束

- 采用 Python 3.12+ 模块化单体架构。
- 分析存储优先使用 Parquet + DuckDB。
- 模型基线使用 scikit-learn Logistic Regression；候选模型使用 HistGradientBoosting，并在独立时间验证集进行概率校准。
- 接口层采用 FastAPI，使用 OpenAPI 定义前后端数据契约。
- 界面采用 React + TypeScript + Vite，图表优先使用 Apache ECharts。
- 第一阶段使用 CLI 和 Windows 任务计划程序，不引入 Airflow、Celery 或微服务。
- 技术路线详情见 `.omx/project-memory.json`。
- 以上为目标技术路线，不代表依赖已经安装；新增依赖仍需用户明确同意。

## 数据边界

- 本项目必须可独立开发、测试和运行，不得依赖任何外部项目的目录结构、代码、运行状态及内部数据结构。
- 外部数据只能通过本项目定义并版本化的文件或 API 契约导入；数据源位置必须可配置，测试必须使用项目内夹具。
- 赛后 `payouts` 只能用作标签、结算和回测，不得用作赛前特征。
- 成功抓取长期保存采集元数据、内容哈希和不可变的标准化记录，不长期保存响应正文。
- 抓取失败或解析异常的原始响应只追加、不覆盖，并在隔离区保留 30 天；标准化数据必须保留来源、抓取时间和校验状态。
- 赔率必须保存带时间戳的快照，禁止用最终赔付替代下注时赔率。
- 训练集、验证集和测试集必须按时间切分，禁止随机打散比赛记录。
- 同一比赛及其衍生记录不得跨数据集边界。

## 模型与策略约束

- 第一版优先使用透明的统计基线和表格模型，不盲目使用深度学习。
- 概率模型和投注策略分离：模型估计概率，策略根据赔率、阈值和风险决定是否入选。
- 所有模型必须与艇位频率、第一人气等简单基线在同一测试集比较。
- 评价至少包含 Log Loss、Brier Score、校准误差、命中率、回收率、最大回撤和覆盖率。
- 调参不得接触最终测试集；策略阈值不得根据测试结果反向挑选。
- 禁止马丁格尔、亏损追投或宣称无风险套利。

```powershell

```

若项目尚未配置某项工具，应如实标记“未配置”，不得伪造通过结果。数据或模型变更还应验证：

- 数据契约、主键唯一性、缺失率和时间戳单调性。
- 特征的时间穿越检测。
- 结算逻辑与已知样例一致。
- 回测结果可由相同配置重复生成。
- 训练与测试时间窗口完全隔离。
